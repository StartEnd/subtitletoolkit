For HTML5 video, `VTT` is usually the safest default.

The native `<track>` element expects browser-friendly caption files. In practice, that means WebVTT. If your source file comes from a transcription tool as SRT or from a subtitle editor as ASS, convert a final delivery copy to VTT before testing it in the browser.

## Why VTT works best

- It is designed for web playback.
- Browser players understand it directly.
- It supports the `WEBVTT` structure expected by HTML5 environments.

VTT also avoids a common handoff problem: developers can attach it directly to a video element, while editors can still keep SRT or ASS as the working format before export.

## Why SRT still shows up

SRT is still common because many subtitle tools export it first. That means web workflows often need a conversion step before final playback. If that is your case, go straight to [How to convert SRT to VTT for HTML5 video](/guides/how-to-convert-srt-to-vtt-for-html5-video/).

Do not paste raw SRT into an HTML5 track and expect every browser or player wrapper to repair it. SRT uses comma milliseconds and does not include the `WEBVTT` header. Those differences are small, but they are enough to break strict WebVTT parsers.

## Where ASS fits

ASS is useful while editing, but it is usually not the final browser delivery format. If your source subtitles are ASS, converting to VTT is often the cleanest web step. The broader workflow is covered in [How to convert subtitle files for web players](/guides/how-to-convert-subtitle-files-for-web-players/).

ASS styling will not map cleanly into basic HTML5 captions. Treat the VTT file as a delivery output, not as a perfect styled replica. If visual subtitle styling matters, test the result in the exact player and CSS environment that will ship.

## Practical workflow

1. Keep editing subtitles in the format your workflow prefers.
2. Convert the final browser version to `VTT`.
3. Test the VTT output in the target player.

## Browser playback checklist

- Confirm the file starts with `WEBVTT`.
- Confirm timestamps use dots, such as `00:00:01.000`.
- Serve the file with a valid text response from your site or CDN.
- Check captions in the actual player, not only in a text editor.
- Keep the source SRT or ASS file if the editing workflow will continue later.

If the file still fails after conversion, check [How to fix invalid WebVTT timestamps](/guides/how-to-fix-invalid-webvtt-timestamps/) before debugging the player. For a broader troubleshooting walkthrough, see [Why subtitles do not show in HTML5 video](/guides/why-subtitles-do-not-show-in-html5-video/).
